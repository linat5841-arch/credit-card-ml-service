# A/B тестирование моделей кредитного скоринга

## 1. Цель эксперимента

Цель A/B теста — сравнить две версии модели машинного обучения для прогнозирования дефолта по кредитным картам:

- **Модель v1 (контрольная группа)** — текущая модель (RandomForestClassifier)
- **Модель v2 (тестовая группа)** — новая версия модели (например, с улучшенной настройкой или другой архитектурой)

Задача — определить, улучшает ли новая модель качество прогнозирования дефолта без ухудшения бизнес-показателей.

---

## 2. Разделение трафика

Трафик пользователей делится случайным образом:

- 50% запросов → модель v1 (контроль)
- 50% запросов → модель v2 (тест)

Распределение может быть реализовано:
- через API-шлюз
- либо внутри сервиса (рандомизация по request_id)

---

## 3. Длительность эксперимента

Продолжительность A/B теста определяется исходя из объёма данных и статистической значимости.

Рекомендуемый период:
- от 2 до 4 недель
- либо до достижения достаточного количества наблюдений (например, ≥ 10 000 запросов)

---

## 4. Метрики оценки

### Основная метрика

- **F1-score для класса дефолта (1)**

Обоснование:
- баланс между Precision и Recall
- критична для задач кредитного скоринга

---

### Дополнительные метрики

- **Recall (чувствительность)**  
  Важно минимизировать пропущенные дефолты (false negatives)

- **Precision**  
  Важно не отклонять слишком много хороших клиентов

---

## 5. Бизнес-метрики

### 1. Снижение ожидаемых финансовых потерь

Можно оценить как:

Loss = FN * Avg_Default_Loss

где:
- FN — количество пропущенных дефолтов
- Avg_Default_Loss — средний убыток на одного клиента

---

### 2. Доля одобренных заявок при фиксированном уровне риска

Показывает, позволяет ли новая модель:
- увеличить количество одобрений
- без роста дефолтов

---

## 6. Статистический анализ

Для сравнения моделей используются:

- **z-test** — для сравнения долей (например, precision, recall)
- **t-test** — для сравнения средних значений метрик

---

## 7. Доверительные интервалы

Для каждой метрики строятся доверительные интервалы (95%):

- если интервалы не пересекаются → различия значимы
- если пересекаются → результат не доказан

---

## 8. Критерий успешности

Модель v2 считается успешной, если:

- F1-score выше, чем у v1 (статистически значимо)
- Recall не ухудшается
- бизнес-метрики не ухудшаются

---

## 9. Влияние на архитектуру

Для поддержки A/B тестирования сервис должен:

- поддерживать несколько версий моделей (v1, v2)
- реализовывать маршрутизацию запросов
- логировать результаты предсказаний

---

## 10. Расширение (опционально)

В будущем можно реализовать:

- автоматическое переключение на лучшую модель
- хранение результатов в базе данных
- интеграцию с системой мониторинга
