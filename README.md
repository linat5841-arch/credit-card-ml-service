# Credit Card Default Prediction ML Service

## Описание проекта

Сервис машинного обучения для прогнозирования дефолта по кредитным картам на основе датасета **Default of Credit Card Clients Dataset** (UCI Machine Learning Repository).

Цель — реализовать production-like ML pipeline:

обучение модели
сохранение модели
REST API для инференса
контейнеризация (Docker)
подготовка к A/B-тестированию

Домен: финансы / кредитный скоринг
Датасет: UCI Default of Credit Card Clients

## Цель проекта

Разработать и внедрить сервис прогнозирования дефолта, готовый к развёртыванию и тестированию разных версий моделей в production-like среде.



## Датасет

Используется датасет **Default of Credit Card Clients Dataset**.

Содержит:
- демографические данные клиентов
- кредитный лимит
- историю платежей
- суммы счетов и платежей

Целевая переменная:
- `default.payment.next.month` — дефолт в следующем месяце



## Используемые модели

В проекте реализованы две версии:

- **v1** — LogisticRegression  
- **v2** — RandomForestClassifier  

Это позволяет реализовать A/B-тестирование.

Используемые модели

В проекте реализованы две версии модели для задачи бинарной классификации дефолта по кредитным картам:

v1 — LogisticRegression
v2 — RandomForestClassifier

Обе модели обучаются на датасете Default of Credit Card Clients Dataset и сохраняются в папке models/:
models/model_v1.pkl
models/model_v2.pkl
Обучение моделей выполняется скриптом:
python models/train_model.py

Выбор версии модели для инференса осуществляется через поле:

- `model_version` — необязательное поле, задаёт версию модели для инференса:
   `v1` — используется по умолчанию
   `v2` — альтернативная версия модели для сравнения



A/B-тестирование

В проекте предусмотрена возможность сравнения двух версий модели через API.

Контрольная группа (A) — модель v1
Тестовая группа (B) — модель v2
Концепция A/B-теста
трафик делится случайным образом в пропорции 50/50
пользователи из группы A получают предсказание модели v1
пользователи из группы B получают предсказание модели v2
Цель теста

Определить, улучшает ли новая версия модели качество прогнозирования дефолта без ухудшения бизнес-метрик.

Основные метрики
F1-score для класса дефолта — основная метрика
Precision — дополнительная метрика
Recall — дополнительная метрика для контроля пропуска дефолтов

Подробный план описан в файле:

Архитектура проекта

Такой подход выбран потому, что:

он проще в разработке и сопровождении;

он снижает инфраструктурную сложность по сравнению с микросервисной схемой.

Бизнес-метрики

Помимо технических метрик качества классификации, для заказчика важны бизнес-метрики:

снижение ожидаемых финансовых потерь за счёт более точного выявления клиентов с высоким риском дефолта;
увеличение доли одобренных заявок при том же уровне риска.

Примеры расчёта:

ожидаемые потери можно оценивать как сумму кредитных лимитов или задолженностей по клиентам, ошибочно отнесённым к безопасным;
качество отбора одобряемых заявок можно оценивать через долю клиентов, которым выдано одобрение при фиксированном допустимом уровне дефолта.
