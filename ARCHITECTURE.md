# Архитектура сервиса прогнозирования дефолта

## 1. Общая архитектура

В рамках проекта реализован ML-сервис, предоставляющий API для прогнозирования дефолта по кредитным картам.

Архитектура включает следующие компоненты:

- ML-модель (RandomForestClassifier)
- слой инференса (model_handler.py)
- веб-сервис (Flask API)
- контейнеризация (Docker)
- оркестрация (Docker Compose)

---

## 2. Выбор архитектуры: монолит vs микросервисы

В данном проекте выбран **монолитный подход**.

### Обоснование:

- простота реализации
- минимальные накладные расходы
- отсутствие необходимости масштабирования
- все компоненты (модель + API) тесно связаны

### Когда нужны микросервисы:

Микросервисная архитектура была бы оправдана при:

- высокой нагрузке
- необходимости масштабирования отдельных компонентов
- наличии нескольких моделей
- разделении команд разработки

---

## 3. Контейнеризация

Сервис упакован в Docker-контейнер, который включает:

- код приложения
- обученную модель
- зависимости (requirements.txt)

Преимущества:

- воспроизводимость окружения
- переносимость
- простота деплоя

---

## 4. Оркестрация

Для управления сервисом используется Docker Compose.

Он позволяет:

- запускать сервис одной командой
- управлять конфигурацией контейнера
- масштабировать систему при необходимости

---

## 5. Возможности масштабирования

В случае роста нагрузки архитектура может быть расширена:

- вынесение API и модели в отдельные сервисы
- использование балансировщика нагрузки
- горизонтальное масштабирование контейнеров

---

## 6. Использование брокеров сообщений (RabbitMQ)

В production-системе может быть добавлен брокер сообщений, например RabbitMQ.

Он может использоваться для:

- асинхронной обработки батч-предсказаний
- очередей запросов при высокой нагрузке
- логирования и сбора событий
- интеграции с другими сервисами

---

## 7. Логирование и мониторинг

В текущей реализации логирование выполняется через стандартный вывод (stdout).

В production-среде рекомендуется:

- логирование в формате JSON
- использование централизованных систем (например, ELK-стек: Elasticsearch + Logstash + Kibana)

Это позволит:

- отслеживать ошибки
- анализировать поведение модели
- мониторить производительность

---

## 8. Инструменты MLOps

### DVC (Data Version Control)

Используется для:

- версионирования датасетов
- отслеживания изменений данных
- воспроизводимости экспериментов

---

### MLflow

Используется для:

- отслеживания экспериментов
- хранения метрик моделей
- управления версиями моделей

---

## 9. Управление версиями моделей

В системе предусмотрена возможность использования нескольких версий моделей:

- model_v1 — текущая версия
- model_v2 — новая версия

Это позволяет:

- проводить A/B тестирование
- безопасно выкатывать обновления
- быстро откатываться при ошибках

---

## 10. Производственный деплой

В текущем проекте используется встроенный Flask-сервер.

В production рекомендуется:

- использовать Gunicorn или uWSGI
- настроить NGINX как reverse proxy
- обеспечить отказоустойчивость и масштабируемость
